## 1. Architecture Overview

### Before (MySQL)

```
Vue.js Frontend
     │  HTTP
Express.js Backend
     │  mysql2/promise pool
MySQL Database (5 tables)
```

### After (MongoDB)

```
Vue.js Frontend   ← UNCHANGED
     │  HTTP
Express.js Backend  ← app.js rewritten (DB layer only)
     │  monk
MongoDB Database (5 collections)
```

**Nothing in the frontend changes.** No Vue component, no API call, no response shape is altered.

## 3. Database Schema Mapping

### 3.1 announcements

#### MySQL Table

```sql
CREATE TABLE announcements (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  title        VARCHAR(255)  NOT NULL,
  content      LONGTEXT      NOT NULL,
  category     VARCHAR(100)  DEFAULT 'General',
  image_url    VARCHAR(500)  DEFAULT NULL,
  is_featured  TINYINT(1)    DEFAULT 0,
  status       ENUM('draft','published','scheduled') DEFAULT 'published',
  publish_date DATETIME      DEFAULT NULL,
  created_at   DATETIME      DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### MongoDB Collection

```json
{
  "_id":          ObjectId("..."),
  "legacyId":     42,
  "title":        "Grand Opening",
  "content":      "Full text...",
  "category":     "Company News",
  "image_url":    "/uploads/announcements/file.jpg",
  "is_featured":  true,
  "status":       "published",
  "publish_date": null,
  "created_at":   "2024-03-17T10:09:00.000Z",
  "updated_at":   "2024-03-17T10:09:00.000Z"
}
```

#### Field Conversion Notes

| MySQL Field              | MongoDB Field           | Conversion                                                                                         |
| ------------------------ | ----------------------- | -------------------------------------------------------------------------------------------------- |
| `id` INT AI              | `_id` ObjectId          | Auto-generated by MongoDB; `legacyId` preserves the original numeric id for migration traceability |
| `is_featured` TINYINT(1) | `is_featured` Boolean   | `0 → false`, `1 → true`                                                                            |
| `created_at` DATETIME    | `created_at` ISO String | `new Date().toISOString()` — kept as string to preserve identical JSON response shape              |
| `updated_at` DATETIME    | `updated_at` ISO String | Same as above                                                                                      |

### 3.5 events

#### MySQL Table

```sql
CREATE TABLE events (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  event_name   VARCHAR(255) NOT NULL,
  client_name  VARCHAR(255) DEFAULT NULL,
  event_date   DATE         DEFAULT NULL,
  video_url    VARCHAR(500) DEFAULT NULL,
  photos_json  LONGTEXT     DEFAULT NULL,
  slug         VARCHAR(255) UNIQUE DEFAULT NULL,
  created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP
);
```

#### MongoDB Collection

```json
{
  "_id":         ObjectId("..."),
  "legacyId":    3,
  "event_name":  "Tech Summit 2024",
  "client_name": "AfriTech Summit",
  "event_date":  "2024-09-20T00:00:00.000Z",
  "video_url":   "https://youtube.com/...",
```

#### Field Conversion Notes — Most Important Change

| MySQL Field            | MongoDB Field                | Conversion                                                                                                                                                                                                                                                                                      |
| ---------------------- | ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `photos_json` LONGTEXT | `photos` Array               | **Key change**: MySQL stored a JSON string that required `JSON.parse()` on every read and `JSON.stringify()` on every write. MongoDB stores the array natively. The field is renamed from `photos_json` to `photos`. The API response still returns `photos` as an array — identical to before. |
| `slug` VARCHAR UNIQUE  | `slug` String + unique index | Uniqueness enforced by MongoDB unique index instead of SQL UNIQUE constraint                                                                                                                                                                                                                    |

## 4. Relationship Conversion

This application uses **no foreign key relationships** between tables. All five tables are independent entities. The original MySQL schema also defined no foreign keys or join tables.

The only pseudo-relationship is the `photos_json` column on `events` which embedded a JSON array — this is now a native MongoDB array (`photos`), which is actually more idiomatic.

**No Many-to-Many pivot tables exist.** No embedded document nesting was needed.

## 5. Query-by-Query Conversion

Every SQL query in the original `app.js` is listed below with its Monk equivalent.

---

### 5.1 Dashboard Stats — COUNT queries

**Original MySQL**

```javascript
const [[{ total_requests }]] = await pool.query(
  "SELECT COUNT(*) AS total_requests FROM service_requests",
);
const [[{ pending_requests }]] = await pool.query(
  "SELECT COUNT(*) AS pending_requests FROM service_requests WHERE status='pending'",
);
const [[{ unread_messages }]] = await pool.query(
  "SELECT COUNT(*) AS unread_messages FROM contact_messages WHERE is_read=0",
);
```

**MongoDB Monk Equivalent**

```javascript
const total_requests = await serviceRequests.count({});
const pending_requests = await serviceRequests.count({ status: "pending" });
const unread_messages = await contactMessages.count({ is_read: false });
```

**Why**: Monk's `.count(filter)` is the direct equivalent of `SELECT COUNT(*) WHERE`. All 7 dashboard counts are parallelised with `Promise.all()` — actually faster than the original sequential MySQL queries.

---

### 5.2 Recent Activity — SELECT + ORDER + LIMIT

**Original MySQL**

```sql
SELECT id, name, email, service_type AS detail, status, created_at, 'request' AS type
FROM service_requests ORDER BY created_at DESC LIMIT 5
```

**MongoDB Monk Equivalent**

```javascript
const rawRequests = await serviceRequests.find(
  {},
  { sort: { createdAt: -1 }, limit: 5 },
);
// Then map to shape:
rawRequests.map((r) => ({
  id: r._id.toString(),
  name: r.name,
  email: r.email,
  detail: r.service_type,
  status: r.status,
  created_at: r.created_at,
  type: "request",
}));
```

**Why**: Monk `.find(filter, options)` accepts `sort` and `limit`. The `AS type` and `AS detail` aliases from SQL are replicated in the `.map()` call to preserve the exact response shape the frontend expects.

---

### 5.3 SELECT all with optional WHERE filter

**Original MySQL**

```javascript
let query = "SELECT * FROM portfolio";
if (category && category !== "All") {
  query += " WHERE category=?";
  params.push(category);
}
query += " ORDER BY is_featured DESC, created_at DESC";
const [rows] = await pool.query(query, params);
```

**MongoDB Monk Equivalent**

```javascript
const query = category && category !== "All" ? { category } : {};
const rows = await portfolio.find(query, {
  sort: { is_featured: -1, createdAt: -1 },
});
```

**Why**: Dynamic WHERE clause construction is replaced with a conditional filter object. Monk's sort object `{ is_featured: -1, createdAt: -1 }` mirrors `ORDER BY is_featured DESC, created_at DESC`.

---

### 5.4 SELECT single by ID

**Original MySQL**

```javascript
const [rows] = await pool.query('SELECT * FROM announcements WHERE id=?', [req.params.id]);
if (!rows.length) return res.status(404)...
```

**MongoDB Monk Equivalent**

```javascript
const doc = await findById(announcements, req.params.id);
if (!doc) return res.status(404)...

// findById utility:
async function findById(collection, id) {
  if (isValidObjectId(id)) return collection.findOne({ _id: monk.id(id) });
  return null;
}
```

**Why**: The `id` in URL params is now a 24-char hex ObjectId string. The `findById` helper validates the format before querying to avoid a Monk error on malformed IDs.

---

### 5.5 SELECT by slug OR id (events)

**Original MySQL**

```javascript
let query = "SELECT * FROM events WHERE id = ?";
if (isNaN(idOrSlug)) query = "SELECT * FROM events WHERE slug = ?";
const [rows] = await pool.query(query, [idOrSlug]);
```

**MongoDB Monk Equivalent**

```javascript
let event;
if (isValidObjectId(idOrSlug))
  event = await events.findOne({ _id: monk.id(idOrSlug) });
if (!event) event = await events.findOne({ slug: idOrSlug });
```

**Why**: The original used `isNaN` to distinguish a numeric id from a slug. The new code tries ObjectId lookup first, then falls back to slug. This handles both legacy numeric ids (which won't be valid ObjectIds) and new ObjectId-based lookups.

---

### 5.6 INSERT and return new row

**Original MySQL**

```javascript
const [result] = await pool.query(
  'INSERT INTO announcements (title,content,...) VALUES (?,?,?)',
  [title, content, ...]
);
const [newRow] = await pool.query('SELECT * FROM announcements WHERE id=?', [result.insertId]);
res.status(201).json({ data: newRow[0] });
```

**MongoDB Monk Equivalent**

```javascript
const doc = await announcements.insert({ title, content, ... });
res.status(201).json({ data: normalizeDoc(doc) });
```

**Why**: Monk's `.insert()` returns the newly created document immediately — no need for a second SELECT query. `normalizeDoc` adds `id` as a string alias of `_id`.

---

### 5.7 UPDATE dynamic fields

**Original MySQL**

```javascript
const fields = {};
["title", "content", "category"].forEach((f) => {
  if (req.body[f] !== undefined) fields[f] = req.body[f];
});
const setClauses = Object.keys(fields)
  .map((f) => `${f}=?`)
  .join(", ");
await pool.query(`UPDATE announcements SET ${setClauses} WHERE id=?`, [
  ...Object.values(fields),
  req.params.id,
]);
```

**MongoDB Monk Equivalent**

```javascript
const fields = {};
["title", "content", "category"].forEach((f) => {
  if (req.body[f] !== undefined) fields[f] = req.body[f];
});
await announcements.update({ _id: existing._id }, { $set: fields });
```

**Why**: Monk's `$set` operator replaces SQL's dynamic SET clause construction. Only fields present in `fields` are updated — identical partial-update semantics.

---

### 5.8 UPDATE single field (status / read)

**Original MySQL**

```javascript
await pool.query("UPDATE announcements SET status='published' WHERE id=?", [
  req.params.id,
]);
```

**MongoDB Monk Equivalent**

```javascript
await announcements.update(
  { _id: existing._id },
  { $set: { status: "published" } },
);
```

---

### 5.9 DELETE

**Original MySQL**

```javascript
const [r] = await pool.query('DELETE FROM announcements WHERE id=?', [req.params.id]);
if (r.affectedRows === 0) return res.status(404)...
```

**MongoDB Monk Equivalent**

```javascript
const existing = await findById(announcements, req.params.id);
if (!existing) return res.status(404)...
await announcements.remove({ _id: existing._id });
```

**Why**: MongoDB doesn't return affected rows from `.remove()`. Instead, we pre-check existence with `findById` (one extra query). This correctly returns 404 if not found — same semantics as `affectedRows === 0`.

---

### 5.10 DISTINCT category values

**Original MySQL**

```javascript
const [rows] = await pool.query(
  "SELECT DISTINCT category FROM portfolio ORDER BY category",
);
res.json({ data: ["All", ...rows.map((r) => r.category)] });
```

**MongoDB Monk Equivalent**

```javascript
const rows = await portfolio.distinct("category");
const sorted = rows.slice().sort();
res.json({ data: ["All", ...sorted] });
```

**Why**: Monk exposes `.distinct(field)` which returns an array of unique values. We `.sort()` manually to replicate `ORDER BY category`.

---

### 5.11 Unread messages filter

**Original MySQL**

```javascript
if (status === "unread") query += " WHERE is_read=0";
else if (status === "read") query += " WHERE is_read=1";
```

**MongoDB Monk Equivalent**

```javascript
if (status === "unread") query = { is_read: false };
else if (status === "read") query = { is_read: true };
```

**Why**: MySQL stored `is_read` as `TINYINT(1)` (0/1). MongoDB stores it as a Boolean (true/false). Both the filter and the insert use `false`/`true`. The JSON response still serialises as `0`/`1`-compatible booleans for the frontend (JavaScript treats both `false` and `0` as falsy, so the frontend's `if (item.is_read)` checks continue to work).

---

### 5.12 Event photos — JSON string vs native array

**Original MySQL (every read)**

```javascript
rows.map((r) => ({ ...r, photos: JSON.parse(r.photos_json || "[]") }));
```

**Original MySQL (every write)**

```javascript
await pool.query("INSERT INTO events (..., photos_json) VALUES (..., ?)", [
  JSON.stringify(photos),
]);
```

**MongoDB Monk Equivalent — no conversion needed**

```javascript
// INSERT
await events.insert({ ..., photos: Array.isArray(photos) ? photos : [] });

// READ — photos is already an array
res.json({ data: { ...event, photos: Array.isArray(event.photos) ? event.photos : [] } });
```

**Why**: This is the biggest quality-of-life improvement. MySQL required serialising the array to a string on every write and parsing it back on every read. MongoDB stores arrays natively, eliminating `JSON.stringify`/`JSON.parse` overhead and the risk of malformed JSON.

## 6. ID Compatibility Layer

### The Problem

MySQL used auto-increment integer IDs: `1, 2, 3, ...`
MongoDB uses ObjectIds: `"507f1f77bcf86cd799439011"`

The frontend accesses `item.id` in many places (Vue templates, API calls like `/api/requests/5`).

### The Solution — `normalizeDoc()`

```javascript
function normalizeDoc(doc) {
  if (!doc) return doc;
  const out = { ...doc };
  out.id = doc._id.toString(); // Adds 'id' as string alias of '_id'
  return out;
}
```

Every API response passes through `normalizeDoc()` or `normalizeDocs()`. The frontend receives both `_id` (ObjectId) and `id` (string) on every object. Since the frontend uses `item.id`, it gets the ObjectId string and sends it back in PUT/DELETE URLs like `/api/requests/507f1f77bcf86cd799439011`. The backend's `findById()` then parses this as an ObjectId.

**Frontend compatibility: zero changes required.**

### Legacy IDs After Migration

The `migrate.js` script stores `legacyId: row.id` on every migrated document. This means:

- Old bookmarked URLs like `/events/3` (numeric) will fall back to slug lookup and still work if the slug exists.
- New items created after migration use ObjectId strings exclusively.

## 7. Step-by-Step Deployment

### Step 1 — Install MongoDB

```bash
# Ubuntu/Debian
sudo apt install mongodb
sudo systemctl start mongodb
sudo systemctl enable mongodb

# Or use MongoDB Atlas (cloud) — just set MONGO_URI in .env
```

### Step 2 — Update dependencies

```bash
cd backend
npm install
# This installs monk and removes mysql2 per the updated package.json
```

### Step 3 — Update .env

```bash
# Remove these lines:
# DB_HOST=localhost
# DB_PORT=3306
# DB_USER=root
# DB_PASSWORD=
# DB_NAME=fourk_production

# Add this line:
MONGO_URI=mongodb+srv://4k-production:4k-production@4k-production.i0zgwjv.mongodb.net/?appName=4k-production
```

### Step 4 — Migrate existing data (if you have live MySQL data)

```bash
# Temporarily install mysql2 for the migration only
npm install mysql2

# Run migration (reads MySQL, writes MongoDB, safe to re-run)
node migrate.js

# Remove mysql2 after migration
npm uninstall mysql2
```

### Step 5 — Start the server

```bash
npm start
# or
npm run dev
```

### Step 6 — Verify

```bash
curl https://4k-production.onrender.com/health
# Expected: {"success":true,"message":"4K Production API running",...}

curl https://4k-production.onrender.com/api/announcements
# Expected: {"success":true,"data":[...]}
```
